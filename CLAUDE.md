# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

ITA Digital is a business management platform (BTP/construction) being built module by module. This repository currently contains **Lot 0 — Socle technique** only: authentication, the role/access model, theming, and generic reusable UI components. No business module (Achat, QHSE, RH forms, Logistique, etc.) is implemented yet — only the structural referential (`Module`/`SousModule`) they will plug into.

Stack: Next.js 16 (App Router) + TypeScript + Tailwind v4 + shadcn/ui + Prisma 6 + Supabase (PostgreSQL + Auth + Storage) + Resend/react-email. Package manager is **pnpm** (not npm/yarn).

## Commands

```bash
pnpm dev            # dev server
pnpm build           # production build (also runs TypeScript checking)
pnpm lint            # ESLint
npx tsc --noEmit     # type-check only, no build

pnpm db:migrate      # prisma migrate dev
pnpm db:seed         # prisma db seed -> runs prisma/seed.ts via tsx
pnpm db:studio       # prisma studio
npx prisma generate  # regenerate the Prisma client after editing schema.prisma
```

There is no test runner configured yet.

### Environment

Copy `.env.example` to `.env.local` and fill in Supabase values (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `DATABASE_URL` pooled/pgbouncer, `DIRECT_URL` direct connection for migrations). `prisma.config.ts` loads `.env.local` explicitly via `dotenv` because the Prisma CLI runs outside of Next.js and does not get Next's automatic env loading. After a schema change, run `db:migrate` then regenerate the client if needed.

`prisma/rls-lockdown.sql` should be run once in the Supabase SQL editor to enable RLS with zero permissive policies on `public.*` — pure defense-in-depth since the app never talks to Postgres through PostgREST (see Data access below).

## Architecture

### Data access: Prisma only, Supabase Auth only for auth

Two separate, non-overlapping systems:
- **Supabase client** (`src/lib/supabase/{client,server,middleware,admin}.ts`) is used **exclusively** for `supabase.auth.*` calls (sign in, sign out, password update, session/cookie refresh). Never call `supabase.from(...)` to read/write `public.*` tables.
- **Prisma** (`src/lib/prisma.ts`, a hot-reload-safe singleton) is the only way application data is read or written, always from Server Components or Server Actions.

Server-side auth checks always use `supabase.auth.getUser()`, never `getSession()` — `getUser()` revalidates the JWT against the Supabase Auth server; `getSession()` trusts the cookie without validation.

The `Utilisateur` (app user) table is linked to Supabase's `auth.users` via a plain scalar column, `Utilisateur.authUserId` (`String @unique @db.Uuid`) — **not** a Prisma relation or a Postgres FK. Prisma's `schemas`/multi-schema config and a shadow `AuthUser` model mapped to `auth.users` were tried and reverted: `prisma migrate dev` treated Supabase's internal auth tables (sessions, MFA, OAuth...) as part of Prisma's managed schema and proposed resetting them as "drift" — dangerous, since that's live Supabase Auth state, not application data. Prisma now only knows about the `public` schema; matching a `Utilisateur` to its Supabase user is always a plain value comparison (`where: { authUserId: user.id }`), never a Prisma `include`/join. No password is ever stored in the `Utilisateur` table — passwords live entirely in Supabase Auth.

### Two-tier route guarding

1. **`src/proxy.ts`** (Next.js 16's replacement for `middleware.ts` — the file convention was renamed) does a coarse, cheap check on every request: refresh Supabase session cookies, redirect unauthenticated requests away from `/dashboard`, redirect authenticated requests away from `/login`. It knows nothing about business rules.
2. **`src/app/dashboard/layout.tsx`** is the single point of truth for fine-grained logic: resolves the Prisma `Utilisateur`, kills the session and redirects if `statut === "INACTIF"` (`requireCompteActif` in `lib/server-actions/guards.ts`), force-redirects to `/premiere-connexion` if `premiereConnexion === true`, then resolves the sidebar's accessible modules.

`src/app/(dashboard)` does **not** exist as a route group — it was deliberately renamed to the real segment `src/app/dashboard/` early on, because a parenthesized route group adds no URL prefix and was silently colliding with the root `/` route from `src/app/page.tsx`. Keep this in mind if you're tempted to reintroduce a `(dashboard)` group: pick a real segment for anything that needs a `/dashboard/...` URL. `(auth)` remains a route group on purpose — `/login` and `/premiere-connexion` are meant to live at the root, not under `/auth/...`.

### Access-control model (module 0 — the referential every future module hangs off)

Two independent axes, deliberately kept separate to avoid a combinatorial explosion of roles:
- `niveauHierarchique` (`DIRECTEUR` | `CHEF_SERVICE` | `AGENT`) — will drive who approves what in future modules' validation workflows.
- `Fonction` (a free-form referential, e.g. "Responsable RH", "Chef de chantier") — drives which `Module`/`SousModule` entries show up in the sidebar.

Key rule from `prisma/schema.prisma`: **every `Module` always has at least one `SousModule`**, including "single-flow" modules like Service Achat (via a synthetic sous-module, e.g. `achat` → `acces`). This is intentional — it keeps `AccesUtilisateur.sousModuleId` non-nullable everywhere, which is what makes the `@@unique([utilisateurId, sousModuleId])` constraint actually enforce uniqueness (Postgres treats `NULL` as always-distinct, so a nullable FK there would silently allow duplicates), and it means sidebar/guard logic never needs a "module without children" special case.

Access resolution (`lib/server-actions/acces.ts`, `getModulesAccessibles`) filters on three things simultaneously: `AccesUtilisateur.actif` (per-user grant), `SousModule.actif` (a global product kill-switch — e.g. RH "Pointage" is seeded `actif: false` and excluded from every Fonction's defaults), and `Module.visibleMenu` (the transversal `authentification-roles` module is `visibleMenu: false` and never appears in the generic menu).

There is deliberately **no separate technical "Admin" role**. The `authentification-roles` module (RH account/access management) is protected the exact same way as any future business module would be, via `requireAccesModule(utilisateurId, moduleCode, sousModuleCode)` in `lib/server-actions/guards.ts` — see `src/app/dashboard/authentification-roles/gestion-comptes-acces/page.tsx` for the reference pattern to copy when adding a protected page. Whether a given Fonction can reach it is just a matter of whether `gestion-comptes-acces` is in that Fonction's `FonctionModuleDefaut` list.

`AccesUtilisateur.estException` marks a per-user manual override made by RH. `propagerModificationFonction()` (`lib/server-actions/user-provisioning.ts`) is the function meant to reconcile every user of a Fonction to a new default value on a given sous-module, skipping rows already flagged `estException: true` — but it has no call sites from any UI today: there is no admin screen yet for editing `FonctionModuleDefaut` (only `prisma/seed.ts` writes that table). This matters because `seedFonctions()`'s upsert of `FonctionModuleDefaut` only edits the *template* row — it never touches existing users' `AccesUtilisateur` grants, and `seedUtilisateursTest()` explicitly skips provisioning for any user that already exists (`if (utilisateurExistant) continue;`). Editing a Fonction's default sous-module list in `seed.ts` and re-running `pnpm db:seed` therefore does **not** propagate to pre-existing users — this is exactly how `dg.test` ended up missing `gestion-comptes-acces` access after the Direction Générale Fonction's defaults were widened to include it. `propagerModificationFonction()` was also fixed alongside this: the original version only `updateMany`'d rows that already existed, so it would have silently no-op'd for a user (like `dg.test`) who never had any row at all for that sous-module — it now `createMany`s the missing rows first (`skipDuplicates: true`), then aligns `actif` on existing non-exception rows. Until a Fonction-defaults admin UI exists, call `propagerModificationFonction(fonctionId, sousModuleId, nouvelleValeur)` yourself (e.g. via a disposable `tsx` script loading `.env.local` with `dotenv`, mirroring `prisma.config.ts`'s own pattern — don't import `user-provisioning.ts` directly into a standalone script, it has `import "server-only"` at the top and will throw outside Next's bundler, same issue already documented for `prisma/seed.ts`/`admin.ts`) any time a Fonction's defaults change and existing users need to pick that up. New users don't have this problem — they get their initial grants copied fresh from `FonctionModuleDefaut` via `initialiserAccesDepuisFonction()` at account-creation time.

**Sidebar visibility never special-cases `niveauHierarchique`.** This was an open question carried over from Lot 0 and was explicitly resolved for Lot 1: `direction-generale` — despite being the module a company director would intuitively expect to always see — is gated by `Fonction`/`AccesUtilisateur` exactly like every other module, with zero branching on `niveauHierarchique === "DIRECTEUR"` in `getModulesAccessibles()`. A user with `niveauHierarchique: DIRECTEUR` but the wrong `Fonction` will not see it; RH is expected to assign the "Direction Générale" `Fonction` to whoever should. `niveauHierarchique` still does exactly what Lot 0 said it would — decide who a validation is addressed to inside a page's content (see `DemandeIndex` below) — it just never decides what shows up in the menu.

### Status/history pattern for future business workflows

`HistoriqueStatut` (in `schema.prisma`) is the generic audit-trail table every future validation circuit (Achat, Congés, AST, etc.) is meant to reuse. It has no FK to business tables — Prisma doesn't support polymorphic relations, so it uses an application-level discriminant (`entiteType` + `entiteId`, indexed together) instead. Write to it via `enregistrerTransition()` in `lib/server-actions/historique.ts`. When you add a business module with a validation workflow, give its table its own `statut` enum/string field and log every transition here rather than inventing a per-module history table.

### Aggregation layer for Direction Générale (`DemandeIndex`)

`DemandeIndex` (schema.prisma) is a **projection of current state**, distinct from `HistoriqueStatut`'s **event log**: one row per business entity (upserted via the same `entiteType`+`entiteId` discriminant, `@@unique` enforced), always reflecting the latest known status — never an append-only history. Every future module with a validation workflow is expected to call both `enregistrerTransition()` and `upsertDemandeIndex()` (`lib/server-actions/demande-index.ts`) on each status change — one for the audit log, one for the DG-level projection — never one without the other.

Convention: a demande counts as "en attente" if `enAttenteValidationDe` (a `NiveauHierarchique`, for circuits that designate a generic role like "Directeur Général") or `enAttenteValidationUtilisateurId` (a specific `Utilisateur`, for circuits like "the requester's direct supervisor") is non-null; "traité" (resolved) once both are null. `upsertDemandeIndex()` always **overwrites** both fields with whatever is passed (defaulting to `null`) rather than merging — so a resolved demande can never keep displaying a stale "waiting on X". This binary signal, not `statutLibelle` (a free string — each module keeps its own status enum/wording), is what drives every DG aggregation query (`src/app/dashboard/direction-generale/{suivi,kpi}/page.tsx`).

The DG pages also need to know which *resolved* demandes were approved vs. rejected (for "montant total validé" in KPI), but `DemandeIndex` deliberately has no structured outcome field for that — it falls back to a keyword heuristic on `statutLibelle` (`tonaliteDepuisStatutLibelle()` in `lib/demande-index-tonalite.ts`: `valid/approuv/accept` → success, `refus/rejet/annul` → danger, `attente` → attention, else neutral), the same one used to color `StatutBadge` in the DG pages. Revisit this if a future module's status wording doesn't match any keyword — it'll silently fall through to "neutre"/excluded-from-validated-total rather than error.

### Theming

CSS custom properties in `src/app/globals.css` mix `oklch()` (shadcn's Nova preset defaults) and raw hex (the ITA Digital brand colors) — that's fine, each CSS variable just holds a color value, the format doesn't need to be consistent across variables. Brand tokens: `--brand-navy` (#002a5c), `--brand-magenta` (#7f0f59), `--brand-green` (#13850b). Five status tones are registered under `--status-{success,info,attention,danger,neutral}` and exposed as Tailwind utilities (`bg-status-danger`, `text-status-info`, etc.) via the `@theme inline` block — use those classes rather than arbitrary-value `[hsl(var(--x))]` syntax. Only `--status-success` maps directly to a brand color; the other four are supplementary since the brand palette only defines three hues. Dark-mode overrides live in a single `.dark { ... }` block — if you add another `.dark` block, later declarations win, so keep it merged rather than appending a second one.

Open Sans is loaded in `src/app/layout.tsx` with `next/font/google` under the CSS variable name `--font-sans` specifically (not `--font-open-sans`), because `globals.css`'s `@theme inline` block maps `--font-sans: var(--font-sans)` — the font's variable name has to match that exactly for the mapping to resolve. `PageHeader` (`components/ui/composed/page-header.tsx`) applies bold titles by convention; use it rather than hand-rolling `<h1>` styling.

The sidebar (`components/ui/composed/sidebar.tsx`) is navy (`--sidebar: var(--brand-navy)`), not the shadcn Nova default — `--sidebar-accent` (active-item highlight) is derived from `--brand-navy` via `color-mix(in oklch, var(--brand-navy), white 15%)` rather than a second hand-picked hex, so it stays in sync if the brand color ever changes. Per-module sidebar icons come from `Module.icone` (a lucide-react export name, e.g. `"Building2"`), populated in `prisma/seed.ts`'s `referentielModules` — `Sidebar`'s `resoudreIcone()` falls back to `Circle` if a module's `icone` is null, so a new module added without an `icone` value silently gets the generic dot rather than erroring; remember to add one when seeding a new module.

### Generic reusable components (`src/components/ui/composed/`)

`StatutBadge`, `NotificationBell`, and the `useNotifier()` hook (`src/hooks/use-notifier.ts`, wraps `sonner`) are intentionally decoupled from any business data — they take generic props (`label`/`tonalite`, `NotificationItem[]`, etc.) and every future business module is expected to map its own status enum / notification source onto these rather than growing module-specific status or notification UI.

**Server Actions that need to show a toast on success must not `redirect()` themselves.** `redirect()` thrown inside a Server Action never gives the calling client component a normal return value to act on, so there's no window to call `useNotifier()` before navigating. The established pattern (`premiere-connexion/actions.ts` + `change-password-form.tsx`, mirroring `login-form.tsx`) is: the action returns `{ erreur } | { succes: true }`, and the **client** component calls `notifier.succes(...)` then `router.push(...); router.refresh()` itself. Keep new Server Actions that trigger a toast on this same shape rather than redirecting server-side.

`FormField` (`components/ui/composed/form-field.tsx`) wraps `Label` + a field + optional helper/error text with consistent spacing and bold labels — pass any single form control as `children` (`Input`, `NativeSelect`, etc.), it never owns the control itself, so it stays compatible with react-hook-form's `register()` spread on the child with no extra indirection. Use it for every new labeled form field rather than hand-rolling `<div className="space-y-2"><Label/><Input/>{error && ...}</div>`.

`NativeSelect` (`components/ui/native-select.tsx`, primitive tier alongside `Input`/`Label`, not in `composed/`) is a thin styled wrapper around `<select>` mirroring `Input`'s exact classes. This project deliberately has no shadcn `Select` usage — use `NativeSelect` for any dropdown rather than introducing the Radix `Select` primitive or hand-rolling another one-off styled `<select>`.

`EmptyState` (`components/ui/composed/empty-state.tsx`) renders the dashed-border "no data" placeholder used at page/section level (`validations-centralisees`, `suivi`, `gestion-comptes-acces`). Pass `bordered={false}` when nesting inside an already-bordered container (e.g. a compact per-item `Card`) to avoid a double frame — don't nest the default bordered variant inside another `Card`. Deliberately *not* used inside `NotificationBell`'s dropdown-popover empty state — that's a small inline context where a dashed-border card would look oversized, not a page-level content area.

Toasts are tinted per type via `--toast-{success,error,info}-{bg,border,text}` tokens in `globals.css` (derived from `--brand-green`/`--brand-magenta`/`--brand-navy` via `color-mix()`, wired into `sonner.tsx`'s `style` prop using Sonner's own `--success-bg`/`--error-bg`/`--info-bg` convention) plus a colored dot icon per type, matching `StatutBadge`'s dot vocabulary. The dot for error/info reuses `bg-secondary`/`bg-primary` (which *are* `--brand-magenta`/`--brand-navy`) rather than `bg-status-danger`/`bg-status-info` — those `--status-*` tokens are an independently-chosen 5-tone palette, not derived from the brand colors, so using them here would mismatch the tinted background's hue.

Buttons use 8px corners (`rounded-md`, which resolves to the existing `--radius-md` token — not a new hardcoded value) and bold text (`font-bold`) in every variant. `destructive` renders solid magenta by reusing `--secondary` (already `var(--brand-magenta)`), not the generic `--destructive` red — keeps the danger/error color language consistent with the toast styling above. `outline` renders a navy border/text (also via existing `--primary` token) rather than a generic gray border, used for secondary actions like "Annuler".

### Account administration (`authentification-roles` / `gestion-comptes-acces`)

Replaces the Lot 0 stub (`dashboard/parametres/acces` no longer exists — moved to `dashboard/authentification-roles/gestion-comptes-acces` to match the `<module-code>/<sous-module-code>` convention every other page follows). Since `authentification-roles` has `visibleMenu: false`, this page has no sidebar entry — reachable only via a conditional "Gestion des comptes" item in `UserMenu`, shown only when `peutGererComptes()` (`lib/server-actions/acces.ts`) returns true for the current user. Protected by `requireAccesModule()` like any module — every exported function in `lib/server-actions/gestion-comptes.ts` re-checks this itself (a Server Action is an independent POST endpoint; the page's own guard doesn't protect it).

**Deletion vs. suspension**: `supprimerUtilisateur()` refuses to delete if the user is referenced by any of the twenty-three FKs to `Utilisateur` that lack `onDelete: Cascade` — `HistoriqueStatut.acteurId`, `DemandeIndex.demandeurId`, `DemandeIndex.enAttenteValidationUtilisateurId`, `AccesUtilisateur.modifieParId`, `Utilisateur.creeParId` itself, `Document.ajouteParId` (Lot 2), `DemandeCarburant.demandeurId`, `Reapprovisionnement.effectueParId` (both Lot 3), `Utilisateur.superieurId`, `DemandeAbsence.employeId`/`superieurId`, `DemandeMission.employeConcerneId`/`initiateurId`, `ReleveActivite.ouvrierId`/`saisiParId` (all Lot 4), `DemandeAppelOffres.initiateurId`, `Projet.chefProjetId`, `PointValidation.cochePar`, `DemandeRHProjet.initiateurId`, `LigneDemandeRHProjet.ouvrierContreProposeId`, `AffectationProjet.ouvrierId`, `DemandeMateriel.initiateurId`, and `Article.creeParId` (all Lot 5, see below) — only `AccesUtilisateur.utilisateurId`, `ProfilEmploye.utilisateurId`, `PieceJointeAppelOffres.demandeId`, `PointValidation.projetId`, and `Notification.destinataireId` cascade (all genuinely "belong entirely to their parent," not an audit record of someone else's action — this distinction is what decides cascade-vs-block throughout the app, not whether the record happens to be historical/closed). Refusing rather than cascading preserves audit trail integrity, uniformly, even for fully resolved (`VALIDEE`/`REFUSEE`) records — no exception is carved out for closed cases. The error message points the admin at suspension (`basculerStatutUtilisateur`) instead. When deletion *is* allowed, the order is always Supabase Auth first, then Prisma — the reverse would leave an orphaned Auth account with no corresponding row, invisible from this page.

**`Utilisateur.creeParId`** is the only self-relation on `Utilisateur` — nullable (the three seed accounts have no human creator), set once at creation in `creerCompteUtilisateur()` from the calling admin's own id, never updated afterward. Because deleting a user who has created other accounts is blocked (see above), the detail page's "Créé par un compte depuis supprimé" fallback text should be unreachable through normal app usage — it exists purely as a defensive display case (e.g. a manual DB fix bypassing the app) rather than a state the UI is expected to produce on its own.

**Welcome email is best-effort, never blocking.** `creerCompteUtilisateur()` sends the temporary password via `resend.emails.send()` (`src/emails/compte-cree.tsx`, modeled on the pipeline-validation placeholder `exemple-notification.tsx`) wrapped in try/catch, checking both a thrown exception and the SDK's own `{ data, error }` return shape — either failure mode is logged server-side and reflected back to the caller as `emailEnvoye: false`, but never turns the whole account-creation call into an error. The account, its `AccesUtilisateur` grants, and the on-screen temporary-password reveal all succeed regardless of email delivery; the client only adapts its toast/dialog wording based on `emailEnvoye`.

*Observed quirk*: `admin.auth.admin.deleteUser()` can transiently fail with `invalid JWT: unrecognized JWT kid ... for algorithm ES256` when called within seconds of that same user's `createUser()` — a Supabase-side propagation lag, not a bug in this code. A retry (from the UI, just click delete again) succeeds. `supprimerUtilisateur()` already fails safe here: it returns the error and does **not** touch the Prisma row, so no inconsistent state results from the failed attempt.

**Individual access changes are a diff, not a snapshot.** `mettreAJourAccesIndividuels()` only upserts (with `estException: true`) the `AccesUtilisateur` rows whose checked state actually differs from the current one — untouched sous-modules keep following `propagerModificationFonction()` as before. Writing every checkbox's state unconditionally would silently mark the whole matrix as manual exceptions, freezing it against future Fonction-level default changes.

**Changing a user's Fonction is a full reset, not a merge.** `changerFonctionUtilisateur()` (`lib/server-actions/user-provisioning.ts`) deletes every existing `AccesUtilisateur` row (including `estException: true` ones) and recreates them from the new Fonction's `FonctionModuleDefaut`. `modifierFonctionEtNiveauUtilisateur()` in `gestion-comptes.ts` gates this behind an explicit confirmation step (`confirmationRequise` response) whenever the user already has exception rows, so an admin can't silently wipe manual overrides by changing a dropdown.

### Document archiving (`assistant-direction` / `archivage-documentaire`, Lot 2)

The first module built on top of the Lot 0 socle, and the first use of Supabase Storage and file upload in this codebase — neither had a precedent before this. Files live in a **private** Storage bucket named `documents` (created once via a disposable script, `public: false`, `fileSizeLimit: "20MB"`, an explicit `allowedMimeTypes` allow-list mirrored in `lib/server-actions/documents.ts`'s `MIME_TYPES_AUTORISES` — PDF, JPEG/PNG/WebP, Word, Excel; no executables/scripts). It is **not** public on purpose: category-based access control (see below) would be meaningless against a guessable public Storage URL. Every read goes through a signed URL (`createSignedUrl`, 5-minute expiry) generated server-side in `listerDocuments()`, and only after that function's own category filtering has already excluded documents the caller isn't allowed to see — an unauthorized user never receives a signed URL for an excluded document, whether or not they could guess its storage path.

`Document.ajouteParId` follows the exact same pattern as `Utilisateur.creeParId` (Lot 1): nullable FK to `Utilisateur`, no `onDelete` cascade, checked as one of the FKs blocking deletion in `possedeDonneesLiees()` (`gestion-comptes.ts` — see below for the current count, now eight with Lot 3).

**Category-based restriction, not module-based.** Everyone with access to the `archivage-documentaire` sous-module sees every category except `RH`, which additionally requires *real* active access (not just Fonction membership) to the `rh` or `direction-generale` module — checked via `possedeAccesModule(utilisateurId, moduleCode)` (`lib/server-actions/acces.ts`), a generalization of `peutGererComptes()` that drops the specific-sous-module-code filter to ask "does this user have any active access within module X at all." This check is applied in three independent places, all server-side, none trusting the client: `listerDocuments()` silently drops `RH` from the query (a forged `?categorie=RH` URL param is simply ignored, not honored or errored — avoids confirming/denying the category's existence to an unauthorized caller), `ajouterDocument()` re-validates before ever touching Storage (the client-side select already omits the "RH" option, but that's cosmetic — a hand-crafted `FormData` submission is still checked), and the category list handed to both the filter `<select>` and the upload form's `<select>` (`categoriesAutoriseesPourUtilisateurCourant()`) never includes `RH` for an unauthorized caller in the first place.

### Gestion Carburant (`carburant` / `depots`, `demande-carburant`, Lot 3)

The first module with a real multi-step validation circuit, and the first real producer of `DemandeIndex`/`HistoriqueStatut` rows — both were built in Lot 1 for the Direction Générale aggregation pages but sat completely unfed until now (confirmed by grep before this lot: zero callers of `upsertDemandeIndex()`/`enregistrerTransition()` anywhere).

**`Vehicule` is deliberately minimal** (immatriculation, type, monthly quota only) — it exists solely so this lot has somewhere to hang a per-vehicle quota. The future Lot 6 (Logistique) is expected to **extend** this model (assignment, history, richer status) rather than replace it — don't casually redesign it away when building Lot 6, migrate/enrich in place.

**Two-step circuit, in-quota vs. hors-quota**: `creerDemandeCarburant()` always starts a demande at `EN_ATTENTE_LOGISTIQUE`. It computes the vehicule's cumulative *validated* litres for the current calendar month once, at creation, and freezes it in `DemandeCarburant.cumulMensuelAvant` — this decides whether the demande is hors-quota, and that decision is **never recomputed later** (matches the cadrage document's own description of this calculation as part of demande creation, not validation). Known accepted simplification: only already-`VALIDEE` demandes count toward the monthly cumulative, so several pending demandes for the same vehicule don't block each other before any of them is actually validated. In-quota demandes resolve directly to `VALIDEE` at the Logistique step; hors-quota demandes move to `EN_ATTENTE_DG` and require a second, separate DG validation before becoming `VALIDEE`.

**Stock-check-at-creation is a courtesy filter, not the real guarantee.** `creerDemandeCarburant()` checks `Depot.quantiteStockLitres` up front purely to reject obviously-doomed requests early. The actual guarantee against overdraft is an atomic conditional update at the moment of final validation (`validerParLogistique()`/`validerParDG()`, inside an **interactive** `prisma.$transaction(async (tx) => ...)`): `depot.updateMany({ where: { id, quantiteStockLitres: { gte: quantite } }, data: { decrement } })`, checking `count === 0` to detect a stock shortfall that appeared between creation and validation and abort cleanly. Never decrement via a bare `update`/`increment` on this field for the same reason.

**Logistique approval routes to one specific person, not a broadcast role — a real limitation, not an oversight.** `DemandeIndex.enAttenteValidationDe` only supports broadcasting to a `NiveauHierarchique` bucket (DIRECTEUR/CHEF_SERVICE/AGENT); there is no mechanism to broadcast to an arbitrary `Fonction`. The DG step correctly uses that broadcast (`enAttenteValidationDe: "DIRECTEUR"`) since any Directeur *with real `direction-generale` access* can act on it. The Logistique step has no equivalent bucket, so `resoudreResponsableLogistique()` (`lib/server-actions/carburant.ts`) resolves it to the single most-senior **active** (`Utilisateur.statut: ACTIF`, not just an active `AccesUtilisateur` row) user holding `carburant`/`depots` access, and targets them directly via `enAttenteValidationUtilisateurId`. This does not generalize to multiple simultaneous Logistique responsables — revisit if a future lot needs that. If no eligible responsable can be resolved at all, `creerDemandeCarburant()` refuses the creation outright rather than writing a `DemandeIndex` row with both validation fields null, which would silently misreport a genuinely-stuck demande as "traitée" in Suivi/KPI.

**Approval guards are real module access, never `niveauHierarchique` used as an authorization check.** `validerParDG()` is gated by `possedeAccesSousModule(id, "direction-generale", "validations-centralisees")`, not `utilisateur.niveauHierarchique === "DIRECTEUR"` — the latter is used correctly elsewhere only as the *routing value* written into `enAttenteValidationDe`, never as an authorization check anywhere in this codebase. Reusing it as a guard would let any Directeur in any department approve fuel requests regardless of whether they have any carburant/DG access at all, bypassing `AccesUtilisateur` entirely — the one thing this project's access model has never done. `peutValiderLogistique()`/`peutValiderDG()` and the new generic `possedeAccesSousModule(utilisateurId, moduleCode, sousModuleCode)` (`acces.ts`, a parameterized sibling to `peutGererComptes()`) are the only sanctioned guards for this circuit.

**The Responsable Logistique/Carburant Fonction deliberately has ONLY `depots` access, not `demande-carburant`.** Giving it both would let a single person submit a fuel request and then approve their own, silently defeating the only check an in-quota request has. "Chef de chantier" was deliberately given `demande-carburant` instead, as a clean, dedicated demandeur persona — distinct from `Responsable RH`'s already-flagged-as-buggy blanket catch-all grant, which incidentally also includes both carburant sous-modules and should not be treated as a real permission source for this module.

**The `demande-carburant` page has a combined guard, not a single `requireAccesModule` call** — `requireAccesCarburantQuelconque()` (`carburant.ts`) admits anyone who can submit, validate at Logistique, or validate at DG, since a Logistique-only responsable (no `demande-carburant` access, by the design decision above) still needs to reach this page to see and act on their "à valider" queue. Every server action called from that page still independently re-derives and re-checks the caller's specific permission — the page-level combined guard only decides who can load the page at all, never what they're allowed to do on it.

### DARH / Ressources Humaines (`rh` module, Lot 4)

Four of the five seeded `rh` sous-modules are built here — `creation-profil`, `conge-permission`, `releve-activite`, `mission`. `pointage` stays `actif: false` and untouched, deliberately deferred.

**`Utilisateur.superieurId`** is a deliberately minimal supervision model — one level only, not a full org chart, no separate "manager chain" concept. It's set from the new "Profil RH" card on the existing user detail page (`gestion-comptes-acces/[utilisateurId]`), even though the field lives on `Utilisateur` and the rest of that card's data (`ProfilEmploye`) doesn't — `mettreAJourProfilRH()` updates both in one transaction, since from a product perspective "who's your supervisor" belongs in the same place as the rest of someone's employment profile.

**A congé/permission request is refused at creation if the routing target can't actually be reached — checked two ways, not one.** `creerDemandeAbsence()` refuses outright if `employe.superieurId` is `null` (same "refuse rather than fabricate a default" principle as Lot 3's `resoudreResponsableLogistique()`), *and* separately refuses if the designated `superieur` lacks `rh`/`conge-permission` access. The second check exists because `superieurId` — unlike `resoudreResponsableLogistique()`/`resoudreResponsableRH()`, which only ever resolve to someone who's *proven* they hold the relevant access — is a free assignment RH could point at anyone, including someone who can never reach the page to act on it. Without this check, a congé request could become permanently stuck the same way Lot 3 was explicitly designed to prevent for Logistique.

**Congé approval authorization is literally "are you the target of the routing," no sous-module gate on top.** `validerAbsence()`/`refuserAbsence()` check `utilisateur.id === demande.superieurId` and nothing else — unlike Logistique/DG in Lot 3 (broadcast-style: any holder of a given sous-module can act), a supervisor relationship is inherently a named, one-to-one routing, so the routing target *is* the authorization. `superieurId` is frozen onto the `DemandeAbsence` row at creation from `employe.superieurId` — stable even if the employee's supervisor changes later, same "freeze the routing decision" reasoning as `DemandeCarburant.cumulMensuelAvant`.

**Congé/permission is scoped as personal data, not shown to everyone with sous-module access like carburant.** The main list on `conge-permission` is "Mes demandes" (own submissions) + "À valider" (routed to the viewer as `superieurId`) — direct precedent already exists in this app for treating RH-adjacent personal data as more restricted than operational data (`Document`'s `RH` category, Lot 2, requires real `rh`/`direction-generale` access beyond generic archive access). Since congé doesn't route through `enAttenteValidationDe` the way mission does, RH has no organic oversight into it otherwise — a read-only "Toutes les demandes" section, gated by `creation-profil` access (the same "is RH" signal used everywhere else in this lot), exists specifically to give RH that visibility without opening the sub-module up to everyone.

**Solde de congés decrements with the same atomic-conditional-update pattern as carburant's stock**, not a bare `decrement` — `validerAbsence()` uses an interactive transaction with `profilEmploye.updateMany({ where: { utilisateurId, soldeConges: { gte: jours } }, data: { decrement } })`. The same employee submitting two overlapping congé requests before either is validated is a realistic race (double-click, two tabs), not just a theoretical one — same shape of bug as the depot stock race, smaller blast radius, same fix.

**Mission and relevé d'activité validators are gated by real access to `creation-profil`**, never by `Fonction`/`niveauHierarchique` — direct reuse of the pattern Lot 3 established with `depots`-access-as-Logistique-signal. `peutValiderRH()`/`resoudreResponsableRH()` (`lib/server-actions/rh-partage.ts`) are shared between `rh-missions.ts` and `rh-releves.ts` rather than duplicated. Both sub-modules route their pending step to one resolved person via `enAttenteValidationUtilisateurId` (same "refuse if nobody eligible" rule as `resoudreResponsableLogistique()`), not a broadcast bucket — there's no `NiveauHierarchique` value for "RH."

**Demande de mission terminates at `VALIDEE_RH` for this lot — not a bug, not left hanging.** The cadrage document describes a follow-on payment step through DFC, but that module doesn't exist yet (Lot 8). `validerMission()` sets both `DemandeIndex` validation fields to `null` (i.e. "traitée") at `VALIDEE_RH`, with an explicit code comment noting the future Lot 8 will pick missions back up at that exact status for payment. Do not add any logic that keeps a mission "pending" waiting for a DFC role that doesn't exist — that would recreate the exact stuck-demande failure mode this lot's other guards were built to avoid.

**Relevé d'activité was fully functional in Lot 4, not a placeholder — only its empty state referenced the Lot 5 dependency at the time.** Chef de chantier submitted real jours-travaillés against a free-text `projetLibelle` field (no real `Projet` referential existed yet), RH validated for real, and it was wired to `DemandeIndex`/`HistoriqueStatut` like every other real circuit in this app. **Lot 5 migrated `projetLibelle` to a real `projetId` FK** (see the Direction Technique section below) — this paragraph is kept for historical context on why the field existed in that shape at all.

**`ProfilEmploye` cascades on user deletion, unlike every other FK this lot adds.** It's the same category as `AccesUtilisateur.utilisateurId` — "belongs entirely to the user," not an audit record of someone else's action — so it's exempted from the block-always policy the other 7 new FKs follow. In practice this rarely matters: an employee with a completed profile who's also submitted a congé/mission is already blocked from deletion by those non-cascading FKs before Prisma ever reaches the profile's cascade.

### Direction Technique (`direction-technique` module, Lot 5)

The largest lot to date — five sub-modules, the first real `Notification` persistence (see below), and the `ReleveActivite` migration referenced above.

**`peutValiderDG`/`peutValiderDirectionGenerale` — shared, not duplicated.** `carburant.ts`'s `peutValiderDG()` and this lot's DG-validation guard for appel d'offres were the exact same 3-line check under two names. The check now lives once, as `peutValiderDirectionGenerale()` in `acces.ts`; `carburant.ts`'s `peutValiderDG()` is now a one-line delegate to it, kept only so `carburant.ts`'s existing call sites don't need renaming. This is the pattern going forward: the first occurrence of a check can live wherever it's first needed, but the *second* occurrence should always trigger promoting it to a shared location (`acces.ts` for generic access checks, a per-domain `-partage.ts` file for domain-specific ones like `rh-partage.ts`) rather than tolerating a second independent copy.

**First real `Notification` table.** `NotificationBell` (Lot 0) existed since the very beginning with zero backing data (`notifications={[]}` hardcoded in `Topbar`) — its own doc comment already anticipated "les futurs modules alimenteront cette liste via le même pattern (entiteType/entiteId) que HistoriqueStatut." This lot builds that: `verifierEtCreerAlertesEcheance(utilisateurId)` (`gestion-projet.ts`) is a **pull-based** check, not a scheduled push — this project has no cron/scheduled-task infrastructure anywhere, and this lot doesn't introduce any. It runs on every `dashboard/layout.tsx` render (the established place for cheap per-navigation checks — `getCurrentUtilisateur`/`getModulesAccessibles`/`peutGererComptes` already run there), scoped to an indexed query for the current user only (as chef de projet OR as their `superieurId`), and `createMany({ skipDuplicates: true })`s against the `@@unique([destinataireId, entiteType, entiteId])` constraint — idempotent, and never flips an already-`lu: true` row back to unread. `Notification` is **not** subject to the "refuse rather than fabricate routing" rule that governs `DemandeIndex` — it isn't part of that read model, has no "traitée" semantics to corrupt, so an absent `superieurId` on a chef de projet just means fewer people get notified, never a blocked `Projet` creation.

**Deadline alerts go to the chef de projet AND their `superieurId`** (the cadrage document names both explicitly — not superieur-only). `Projet` creation is never blocked by a missing `superieurId`; the alert mechanism just silently notifies one fewer person.

**`LigneDemandeRHProjet` — validation always names a real ouvrier, immediately.** This is forced by the confirmed product decision that a validated line creates a binding `AffectationProjet`, and `AffectationProjet.ouvrierId` is required — there's no "approve the abstract need, assign a person later" path in this circuit. `validerLigneDirectement(ligneId, ouvrierId)` creates the affectation with the *requested* competence/taux immediately; `contreProposerLigne(...)` proposes a *different* ouvrier/competence/taux and waits on the demande's `initiateurId` (the DT who submitted it) to `accepterContreProposition`/`refuserContreProposition` — acceptance is the actual "validation finale" moment and is what creates the affectation, using the contre-proposed terms. A DT refusing a contre-proposition is terminal (`REFUSEE`) — no renegotiation loop, matching every other refusal path in this codebase (`DemandeCarburant`, `DemandeAbsence`, `DemandeMission` are all equally terminal-on-refusal). `AffectationProjet.ligneDemandeId` is the required, unique side of that relation (not a nullable FK on the line) since every affectation has exactly one origin — this lets the affectation and its origin line stay consistent from a single `create`, no create-then-update pair. `HistoriqueStatut`/`upsertDemandeIndex()` for this circuit are tracked at the **line** grain (`entiteType: "LigneDemandeRHProjet"`), not the parent `DemandeRHProjet` — a demande with 3 lines can have 3 different statuses simultaneously, which a single parent-level `DemandeIndex` row couldn't represent.

**"Ajouté au relevé d'activité des ouvriers" means a binding `AffectationProjet`, not just an audit trail entry.** `creerReleveActivite()` (Lot 4, `rh-releves.ts`) now requires `AffectationProjet.findFirst({ ouvrierId, projetId })` to exist before allowing a relevé — a real gate, not informational. `AffectationProjet` is **deliberately permanent for this lot** — no `statut`/`dateFin`, so an ouvrier reassigned off a project later remains technically eligible to have relevés logged against it indefinitely. This is a known, explicit simplification (unlike `Vehicule`'s inert-referential minimalism, this one gates a live check, so it's worth restating here rather than leaving it to be discovered) — a future lot should add a lifecycle to `AffectationProjet` if this becomes a real problem.

**`Materiel` (4.4) and `Article` (4.5) are two distinct referentials, not the same catalog under two names.** `Materiel` is borrowable equipment inventory for the future Lot 6 (Logistique) to extend — designation + availability only, same minimal-and-extensible spirit as `Vehicule` (Lot 3). `Article` is a purchasing/pricing catalog for the future Lot 7 (Achat) to extend on the *same table* — `fournisseur`/`prix` columns exist now (nullable) but no form field sets them yet; Lot 7 adds editing capability on this row, it does not get its own new table.

**`demande-logistique` (4.4) is a real, complete circuit — not a stripped-down version — but its behavior in the *current* seed is skewed by the already-documented "Responsable RH" catch-all bug, and Lot 6 did NOT fix this despite introducing a real Logistique Fonction.** No sous-module in the already-seeded `logistique` (Service Logistique) referential is named anything like "validation" — `magasins` (warehouses/stock) was picked as the provisional "responsable Logistique" signal, the closest structural analog to `carburant/depots` in Lot 3. `resoudreResponsableLogistiqueMateriel()` (`demande-materiel.ts`) resolves the *oldest* active user with `logistique/magasins` access. Lot 6 (Livraison A) seeds a real, dedicated "Responsable Service Logistique" Fonction granting exactly that access — but "Responsable RH"'s `tousLesSousModulesSauf(["pointage"])` catch-all already included `magasins` since Lot 3/seed's original account creation, and `rh.test` predates every Lot 6 test account. Seniority-based resolution therefore still routes to `rh.test`, not to `responsable-logistique.test`, even after Lot 6. This is the same pre-existing, already-flagged "connu, non corrigé" issue resurfacing yet again — deliberately **not** patched mid-lot (narrowing RH's catch-all is a standing decision this project has consistently deferred, not something to slip in as a side effect of an unrelated lot). The circuit's own logic (model, guard, resolver, validate/refuse actions, `DemandeIndex`/`HistoriqueStatut` wiring) is correct and requires zero further code changes — only ever narrowing RH's seed grant (or deleting/deactivating `rh.test`) would let the dedicated Fonction actually win the resolution.

### Service Logistique (`logistique` module, Lot 6 — Livraison A : stock)

Livraison A covers 6 of the 14 already-seeded `logistique` sous-modules — `magasins`, `fiche-inventaire`, `flux-sortie`, `flux-entree`, `inventaire-periodique`, `seuil-alerte` — plus unblocking Lot 5's `DemandeMateriel`. A second delivery ("Livraison B" — `Vehicule` extension, `suivi-vehicules-engins`, `pieces-admin-vehicules`, `suivi-conso-carburant`, `demande-transport`, both inspection sous-modules, `bon-sortie`, `tableaux-suivi`) is intentionally deferred. Unlike every prior lot, this one is built against a real internal document ("Workflow_Gestion_Inventaires_ITA") the user supplied mid-planning — 4 real magasins, 6 real catégories (not the 8 a document-less first draft had invented), 6 real unités de mesure, and the exact DMS/BSM/BEM/FIC form fields — rather than the cadrage document CLAUDE.md otherwise only references indirectly.

**`Magasin` is a real referential, not just an access anchor like `carburant/depots` was.** Four seeded rows (`MAG-01` Central/Abidjan, `MAG-02` Bouaké, `MAG-03` Botro, `MAG-04` Base vie), each with a nullable `responsableId` — the **Chef Magasin**, a named individual per magasin. This is the same "named routing, authorization = is-the-target" pattern as `DemandeAbsence.superieurId` (Lot 4): `verifierStockDMS()` checks `utilisateur.id === demande.magasin.responsableId`, not a sous-module access flag. `creerDemandeMiseADisposition()` refuses outright if the target magasin has no `responsableId` — same "refuse rather than fabricate routing" rule used everywhere else.

**Two distinct Logistique personas, deliberately separate Fonctions.** "Responsable Service Logistique" (the **Logisticien**) gets all 6 Livraison-A sous-modules — a strategic, cross-magasin role that decides on DMS and validates BEM/inventaire sessions, resolved via the same seniority-based `accesUtilisateur.findFirst(... orderBy dateCreation asc)` pattern as `resoudreResponsableLogistique()` (Lot 3). "Chef de Magasin" (new) gets only the 3 operational sous-modules (`flux-sortie`/`flux-entree`/`inventaire-periodique`) — no `magasins`/`fiche-inventaire`/`seuil-alerte`, which stay Logisticien-only. A Chef de Magasin's actual authority to verify a DMS comes from `Magasin.responsableId`, not from this Fonction — the Fonction only grants page access. Neither role overlaps with "Responsable Logistique/Carburant" (Lot 3, scoped to `carburant/depots` only) — three similarly-named but non-overlapping Logistique personas now exist in this app, by design.

**`Materiel` (Lot 5) is extended in place with a real fiche-article, not replaced.** New nullable fields — `reference` (unique), `categorieId`, `uniteMesureId`, `magasinId`, `emplacement`, `fournisseurHabituel`, `quantiteStock`, `dateCreationStock`, `seuilAlerte`, `stockSecurite`, `quantiteReapproStandard`, `delaiFournisseurMoyenJours` — all nullable because the Lot 5 seed row ("Bétonnière 350L", a borrowable single-item piece of equipment, not a magasin-tracked stock article) keeps every one of them `null` forever. `quantiteStock` is a persisted running balance, same discipline as `Depot.quantiteStockLitres` (Lot 3) — never touched by a generic edit form, only by `MouvementStock`-producing actions. **A given désignation held in two different magasins is two separate `Materiel` rows**, each with its own `reference`/stock/seuil — `magasin` is a field on the fiche article itself, not a many-to-many stock join, matching the source document exactly.

**`CategorieMateriel`/`UniteMesure` are referential tables, not enums** — the same reasoning as `Module`/`SousModule`: a Prisma enum needs a schema migration to add or rename a value; these tables are corrected with a plain `pnpm db:seed` re-run. Seeded with the real 6 categories (Petit matériel, Mobilier, EPI, Fourniture, Entretien, Consommable — **not 8**; Véhicules/Engins are deliberately excluded, tracked separately via `Vehicule` in the future Livraison B) and the real 6 unités (Pièce, Kg, Litre, Mètre, Boîte, Autre).

**`MouvementStock` is one journal table for three kinds of movement** (`ENTREE`/`SORTIE`/`AJUSTEMENT`), never written to directly by a form — only by the validation step of a BSM, a BEM, or the clôture of an inventory session. The atomic pattern is reused verbatim from `carburant.ts` (Lot 3): increments (`flux-entree` validation) are a plain batched `$transaction([...])`; decrements (`flux-sortie` approval) are an **interactive** `$transaction(async (tx) => ...)` with a conditional `updateMany({ where: { quantiteStock: { gte } } })`, checking `count === 0` to reject an over-draft — never a bare `update`/`decrement`.

**Flux Sortie Stock is a 3-actor circuit (DMS → BSM), not a single action — this is a genuine escalation in circuit complexity from the original plan draft, driven directly by the source document's form fields.** `creerDemandeMiseADisposition()` (any user with `logistique/flux-sortie` — today only "Chef de chantier", per Lot 3's seed) starts a `DemandeMiseADisposition` at `EN_ATTENTE_VERIFICATION`, routed via `enAttenteValidationUtilisateurId` to `magasin.responsableId`. `verifierStockDMS()` (the Chef Magasin, authorized by being the target) moves it to `EN_ATTENTE_DECISION`, routed to the resolved Logisticien. `deciderDMS()` (Logisticien) branches four ways: `APPROUVEE` creates a `BonSortieMagasin` and atomically decrements stock per ligne inside one interactive transaction; `TRANSFERT_INTER_MAGASIN` decrements the source and increments a same-designation `Materiel` row in the target magasin if one exists (refuses with an explicit error otherwise — no auto-creation of a fiche in the target magasin); `DEMANDE_ACHAT_DECLENCHEE` creates/reuses a `DemandeReapprovisionnement` via the same helper `seuil-alerte.ts` exports for its own passive detection; `REFUSEE` is terminal with a required `motifRefus`, matching every other refusal path in this app. Every branch resolves the `DemandeMiseADisposition` itself (`DemandeIndex` → both validation fields null, "traité") even when the *underlying material need* isn't actually resolved (transfer/achat) — the DMS's own lifecycle and the downstream fulfillment are tracked as separate concerns, same principle as `DemandeMission` handing off to a future Lot 8 DFC step. `BonSortieMagasin.preparateurId` is auto-attributed to the Chef Magasin who verified the DMS, not a separate "préparation" action — and `confirmerReceptionBSM()` (gated by `utilisateur.id === demande.demandeurId`, same routing-is-authorization pattern) is the final step that closes the loop.

**Flux Entrée Stock (BEM) is a 2-step circuit where validation, not creation, updates the stock.** `creerBonEntreeMagasin()` is open to anyone with `logistique/flux-entree` access and auto-attributes `receptionneParId` to the creator — recording a delivery isn't a request addressed to anyone, so there's no named-routing refusal case here, unlike DMS. `validerBonEntreeMagasin()` (Logisticien) is what actually increments `Materiel.quantiteStock` and logs the `MouvementStock ENTREE` rows, exactly matching the source document's own description of the workflow.

**`inventaire-periodique` (FIC) applies discrepancies as a direct SET, not an increment/decrement.** `cloturerSessionInventaire()` sets `Materiel.quantiteStock = ligne.quantitePhysique` for every non-zero-écart line inside a plain batched transaction, plus one `MouvementStock AJUSTEMENT` per adjusted line — a physical count is a correction to ground truth, not a race-prone delta, so no `gte`-guarded conditional update is needed here the way it is for a sortie. **Known, explicitly flagged simplification**: a `flux-sortie` happening between the physical count and the session's clôture would be silently overwritten by the stale counted value — acceptable for an infrequent, scheduled operation, not a high-concurrency path like `demande-carburant`.

**`seuil-alerte`/`DemandeReapprovisionnement` — the hardest design call in this lot.** Unlike `DemandeMateriel` (Lot 5), which could legitimately resolve "nobody" against an already-seeded `logistique/magasins` sous-module and refuse, there is **no** Achat module/Fonction/sous-module seeded anywhere — not even a stub — so there's no access-check to correctly resolve to "nobody" against. Writing a `DemandeIndex` row with both validation fields null here would always read as "traité" (per this file's own binary-signal rule), silently misreporting a genuinely-unresolved reapprovisionnement need. The fix: `DemandeReapprovisionnement` is a real, audited (`enregistrerTransition()`) entity, created automatically — either by `verifierEtCreerDemandesReapprovisionnement()` (a **page-level** check on `seuil-alerte`'s own load, not a `dashboard/layout.tsx` global check like Lot 5's deadline alerts, because a stock threshold isn't addressed to a specific person the way a project deadline is — running it for every user on every navigation would be both wasteful and semantically wrong) or by `deciderDMS()`'s `DEMANDE_ACHAT_DECLENCHEE` branch (same underlying helper, `creerOuReutiliserDemandeReapprovisionnement()`, idempotent via `@@unique([materielId, statut])`) — but **deliberately never calls `upsertDemandeIndex()`**. It's surfaced only via a dedicated section on the `seuil-alerte` page itself, explicitly labeled "en attente du module Achat (Lot 7)" — an honest reflection of "nowhere to route to yet," not a fabricated resolution. The TSG-INV tableau-de-suivi columns the source document describes (statut OK/ALERTE, demande d'achat en cours) are computed at read time as extra columns on the existing `fiche-inventaire` listing, not a separate page — `tableaux-suivi` itself stays reserved for Livraison B, where it can combine stock and vehicle summaries; the document's complementary "synthèse mensuelle par magasin" table is deferred there too.

**`magasins` stays intentionally light**, mirroring `carburant/depots`'s role as an access anchor rather than a data owner — `fiche-inventaire` is already the sous-module that owns Materiel CRUD/listing, so `magasins` only aggregates a per-magasin summary (articles suivis, alertes actives, DMS/BEM en attente) plus the Chef Magasin's name.

**`DemandeMateriel` (Lot 5) unblock is seed-only, zero code changes to `demande-materiel.ts` — but the unblock is incomplete in practice, see the Lot 5 section above.** The new "Responsable Service Logistique" Fonction grants real `logistique/magasins` access, which is technically all `resoudreResponsableLogistiqueMateriel()` ever needed — but because resolution picks the *oldest* active user with that access, and `rh.test`'s pre-existing catch-all already had it since Lot 3's original seed, `rh.test` still wins over `responsable-logistique.test`. Not patched here, consistent with this project's standing policy of never narrowing RH's catch-all as a side effect of an unrelated lot.

### Service Logistique (`logistique` module, Lot 6 — Livraison B : véhicules, engins, transport)

Covers the 8 remaining `logistique` sous-modules: `suivi-vehicules-engins`, `pieces-admin-vehicules`, `suivi-conso-carburant`, `demande-transport`, `inspection-vehicule`, `inspection-engin`, `bon-sortie`, `tableaux-suivi`. **Source of truth is the original cadrage document's sections 6.7–6.14, recovered directly from this session's very first transcript message** (not summarized or reconstructed) — every circuit, status enum, and field list below traces to that document, with two deliberate exceptions already decided against real fleet Excel data before this lot: `Vehicule.etat` uses 3 real-world values (`OK`/`PANNE`/`HORS_SERVICE`), not the document's 5 (`En service | En réparation | En standby | Réformé | Cédé`), and `TypeVehicule` gains `ENGIN` as a 3rd value rather than tracking engins separately.

**`Vehicule` extended in place, not replaced** — exactly as the Lot 3 doc comment anticipated. New fields: `numeroInterne`/`marque`/`modele`/`annee`/`dateEntree`/`modeAcquisition` (identification), `etat`/`chauffeurActuelId`/`chantierActuel`/`compteurActuel`/`dateAffectation` (affectation courante), `dateDerniereVerificationInventaire`/`verificateurDerniereVerificationId`/`etatConstateDerniereVerification` (dernière vérification), and a **separate dimension** `dateSortieDefinitive`/`motifSortieDefinitive`/`valeurResiduelle` for cession/réforme — deliberately not folded into `etat`, since a vehicle can be `HORS_SERVICE` (temporarily broken) without being definitively retired, and vice versa. The document's own "alerte DFC pour mise à jour des immobilisations" on definitive exit is **not implemented** — DFC (Lot 8) doesn't exist, same "refuse/stay explicit" principle as Achat/QHSE elsewhere in this module: the field is recorded, no fabricated downstream notification.

**`HistoriqueMouvementVehicule` is a dedicated table, not `HistoriqueStatut`** — same reasoning as `MouvementStock` (Livraison A): a changement d'affectation has a real avant/après shape (état avant/après, chantier avant/après) that the generic table doesn't capture well. Written alongside the `Vehicule` update in the same batched `$transaction([...])` — no conditional guard needed, changing a status isn't a concurrent operation to protect.

**`TypePieceAdministrative`/`PointInspection`/`DocumentInspection`/`ItemChecklistTransport` are all referential tables, not enums** — same reasoning as `CategorieMateriel`/`UniteMesure` (Livraison A): corrigible without a migration. `TypePieceAdministrative` (8 rows) and the 10+4 `DocumentInspection` rows and 16 `ItemChecklistTransport` rows are **real content** from the Excel "Contrôle réglementaire" files and the cadrage document. The 30+27 `PointInspection` rows are **partially placeholder** — the document only enumerates 8 of the 27 engin points ("chenilles, vérins, godet, dents du godet, gyrophare, bip de recul, flexibles, freins") and none of the 30 véhicule points; those 8 are seeded verbatim, the remaining 49 are plausible BTP-fleet content clearly flagged in `seed.ts`, correctable without a migration since it's a table.

**`verifierEtCreerAlertesExpirationPieces()` resolves its Logisticien recipient via the exact same seniority-based `accesUtilisateur.findFirst(... orderBy dateCreation asc)` query as everywhere else in this module — and inherits the exact same already-documented "Responsable RH catch-all" skew.** Confirmed during this lot's own verification: the alert routed to `rh.test`, not `responsable-logistique.test`, for the same reason `DemandeMateriel`'s resolution does (Lot 5/6A section above) — RH's `tousLesSousModulesSauf(["pointage"])` grant predates every dedicated Logistique test account. Not a new bug, not patched here, same standing policy.

**Demande de Transport (6.10) is a 5-step circuit reusing both Livraison A Logistique personas, no new Fonction.** Demandeur (Chef de chantier, already has `logistique/demande-transport` access from Lot 3's seed) submits → **Responsable Service Logistique** (`peutViserLogistique`, `logistique/magasins` access) visas and affects a `Vehicule` → **the Garage role** (Départ/Retour checklist + compteur + conducteur) is resolved not via a `Vehicule.magasinId` (vehicles don't belong to a magasin, unlike `Materiel`) but via **identity**: `estResponsableDeMagasin(utilisateurId)` checks whether the caller is `Magasin.responsableId` for *any* magasin, deliberately excluding the demandeur (a Chef de chantier who happens to also hold `flux-sortie` access) from validating their own départ/retour — a self-dealing risk the naive "any `flux-sortie` holder" check would have reopened. `DemandeIndex` resolves to "traité" at `enregistrerDepart()`, not at creation — the retour is a physical closing record (`HistoriqueStatut` only), not a fresh approval waiting on someone, a deliberate simplification of the document's fuller "Vérifié (Départ) → En cours → Vérifié (Retour) → Archivé" status chain.

**The two inspection fiches (6.11a/b) are genuinely separate models, not a shared polymorphic shape** — their en-têtes, document counts (10 vs 4), and signature roles ("Remorqueur du véhicule" vs "Transporteur de l'engin") really differ in the source document. Only the point/document *referential* (`PointInspection`/`DocumentInspection`) is shared, discriminated by `categorie`. **Anomaly handling is asymmetric by design, straight from the document's own wording**: véhicule anomaly → `Notification` to the resolved Logisticien only ("déclenchement **possible** d'une Fiche Non-Conformité QHSE" — QHSE isn't built, so only the real part, the Notification, happens, same "refuse/stay explicit" principle as Achat); engin anomaly → **`Vehicule.etat` is automatically forced to `HORS_SERVICE`** inside the same transaction as the inspection write, because the document states "mise en service bloquée tant que l'action corrective n'est pas faite" for engins specifically — a real gate, not a soft alert, reusing the `etat` field already built rather than inventing a separate blocking flag.

**Bon de Sortie et Transfert (6.12) is a different circuit from `BonSortieMagasin` (Livraison A) despite the similar name — confirmed directly by the user after an initial misread.** `BonSortieMagasin` is the *output* of the DMS approval chain (Chef Magasin verifies → Logisticien decides). `BonSortieTransfert` has no such precondition: a direct triple-signature circuit (Demandeur → Expéditeur-Convoyeur → Responsable de Sortie) for moving any existing `Materiel` — équipement/immobilisation, not a stock-consumption request. Its lines reuse the `Materiel` referential from Livraison A as instructed, and `viserResponsableSortie()` decrements stock via the exact same atomic interactive-transaction pattern (`updateMany({ gte })`, `count === 0` → refuse) and writes to the **same `MouvementStock` ledger** as `flux-sortie` — a `Materiel`'s stock history is coherent regardless of which of the two circuits moved it.

**`suivi-conso-carburant` and `tableaux-suivi` are pure reporting, no new tables.** The former cross-references `DemandeCarburant` (Lot 3) — kilométrage parcouru is derived from the min/max of `DemandeCarburant.kilometrageCompteur` snapshots in the period, since no other odometer history exists for this calculation; ratio computed at read time, never persisted. The latter's "synthèse mensuelle par magasin" is the table explicitly deferred from Livraison A's `fiche-inventaire`; its vehicle-disponibilité view is a simple per-`TypeVehicule` rate (en service vs indisponible), **not** the day-by-day calendar grid the "Suivi Disponibilité" Excel files use — judged out of scope for a synthesis view, revisit if insufficient.

### Service Achat (`achat` module, Lot 7)

The most complex circuit in the project: unlike every prior lot (carburant, congé, mission, appel d'offres, DMS/BSM — all sequential, one validator or one broadcast role at a time), the standard validation step here is **parallel**: the Responsable Achat dynamically selects a subset of DT/RH/DFC/DG who must *all* independently approve, and a single refusal from *any* of them blocks the whole request. Replaces the Lot 0-era synthetic placeholder — `achat/acces` is retired (`actif: false`, same inert-retirement pattern as `rh/pointage`, never deleted, to avoid orphaning the pre-existing RH-catch-all `AccesUtilisateur` row referencing it) in favor of 4 real sous-modules: `demande-achat`, `traitement-achat`, `validations-parallele`, `suivi-demandes`.

**Parallel multi-validator representation — the hard design problem.** `DemandeIndex` only supports one routing target at a time (`enAttenteValidationDe: NiveauHierarchique` OR `enAttenteValidationUtilisateurId: Utilisateur`), so it cannot natively represent "N concurrent approvals, any one refusal blocks all." The fix is `ValidationDemandeAchat`, a child table tracking one row per **role** (`DT`/`RH`/`DFC`/`DG`) selected by the Responsable Achat for that specific demande — never a named individual, matching every other broadcast-style validation in this app (DG, RH): whichever real holder of that role's access acts first claims the row via a conditional `updateMany({ where: { id, statut: "EN_ATTENTE" } })` (count===0 ⇒ "déjà traité par un autre valideur"), the same atomic-guard idiom already used for stock decrements. Role eligibility (`lib/server-actions/achat-partage.ts`) reuses existing signals rather than inventing new ones: DT → `possedeAccesModule(id, "direction-technique")`, RH → `possedeAccesSousModule(id, "rh", "creation-profil")` (the already-established "is RH" signal), DFC → `possedeAccesModule(id, "dfc")`, DG → the existing `peutValiderDirectionGenerale()`. `recalculerStatutDemandeAchat()` (`achat-validations.ts`) runs after every individual decision: any `REFUSEE` row transitions the parent to `REFUSEE`/`etapeBlocage: VALIDATION_PARALLELE` (conditional `updateMany`, count===1 guard); all `VALIDEE` transitions it to `BC_EMIS` and calls `emettreLignesBonDeCommande()` — the same count===1 guard prevents a double-emission race when two roles resolve near-simultaneously.

**The `enAttenteValidationDe: "DIRECTEUR"` sentinel is a deliberate, documented approximation, not a real routing claim.** No single `NiveauHierarchique` bucket or individual can represent "DT+DG pending simultaneously," so the pending-parallel state writes this fixed sentinel purely to preserve `DemandeIndex`'s binary "en attente" signal (the same trap `DemandeReapprovisionnement`, Lot 6, explicitly avoids by never writing to `DemandeIndex` at all — here, unlike there, a real routing destination *does* eventually exist, so the fix is a safe-but-imprecise sentinel rather than opting out entirely). This has a real, harmless effect on `direction-generale/validations-centralisees`, which filters literally on `enAttenteValidationDe === utilisateur.niveauHierarchique` — any `DIRECTEUR`-level tester will see the row surface there during step 4 regardless of whether they hold any achat-relevant access, but real authorization is always independently re-checked per role inside `validerParallele()`/`refuserParallele()`, exactly like every other module's page-vs-action guard split — clicking through with no real access simply leads nowhere actionable.

**Two distinct refusal branches, not one uniform "return to step 3."** `EtapeBlocageDemandeAchat` (`DIRECTEUR` | `VALIDATION_PARALLELE`) records *which* step blocked a `REFUSEE` demande, because the correct resubmission target differs: a Directeur refusal (`resoumettreApresRefusDirecteur()`, `achat-demandes.ts`) returns straight to step 2 — Achat was never involved, so there's nothing there to redo. A parallel-validator refusal (`resoumettreApresRefusValidateurs()`, `achat-traitement.ts`) returns to step 3, since Achat's own terms (price/fournisseur/payment/validator selection) are what's stale. **Resubmission after a parallel refusal does a full reset**: every `ValidationDemandeAchat` row for the demande — including roles that had already said `VALIDEE` before the refusal — is deleted and recreated fresh at `EN_ATTENTE` for whichever roles are reselected, never a partial merge. Same principle as `changerFonctionUtilisateur()`'s "full reset, not merge": if the underlying terms changed, a prior approval on the old terms must not silently carry forward. Verified directly against the live schema (a disposable script, since these server-action files import `acces.ts` which carries `import "server-only"` and cannot be imported into a standalone `tsx` script — same documented limitation as `user-provisioning.ts`): DT and RH validate, DFC refuses → demande `REFUSEE`, DT/RH rows untouched; resubmission → **all four** rows reset to `EN_ATTENTE`, DT/RH included; all four re-approve → `BC_EMIS`.

**`BonDeCommande` — "one open BC per fournisseur," resolved without a raw-SQL partial index.** `ouvertMarqueur: Boolean?` is `true` only while `statut: OUVERT`, reset to `null` (never `false`) once marked `ENVOYE`. `@@unique([fournisseur, ouvertMarqueur])` therefore only rejects a second simultaneous `OUVERT` row per fournisseur — Postgres treats `NULL` as always-distinct in a unique constraint (the same fact `DemandeIndex`'s own binary-signal convention already leans on), so any number of `ENVOYE` rows can coexist for the same fournisseur. `trouverOuCreerBonDeCommandeOuvert()` (`achat-bons-commande.ts`) is a plain find-or-create against this constraint, same idempotent shape as `creerOuReutiliserDemandeReapprovisionnement()` (Lot 6). `emettreLignesBonDeCommande()` groups a demande's lines by each line's own `fournisseur` (set at treatment time, lines within one demande can go to different suppliers) and appends one `LigneBonDeCommande` per line to the relevant open BC — confirmed end-to-end: two entirely separate demandes for the same fournisseur land their lines in the *same* open BC; marking it `ENVOYE` and requesting a third line for that fournisseur opens a fresh BC rather than reusing the sent one. No dedicated `bons-commande` sous-module exists — the BC list/emission/"Marquer Envoyé" actions are gated by `traitement-achat`, the same sous-module as the rest of the Responsable Achat's work.

**Three distinct actors on `DemandeAchat`, deliberately not merged into two.** `demandeurId` (recipient — the real beneficiary, whose `superieurId` drives `directeurDepartementId`), `emetteurId` (owner — often equal to `demandeurId`, but a separate field), `creeParId` (created_by — whoever actually filled the form, can differ from both, same "creator may differ from subject" precedent as `Utilisateur.creeParId`/`Document.ajouteParId`). The department-director step reuses the `DemandeAbsence.superieurId` pattern exactly (Lot 4): `directeurDepartementId` is frozen from `demandeur.superieurId` at creation (re-frozen on Directeur-refusal resubmission, since the supérieur may have changed), creation is refused outright if `superieurId` is null, and separately refused if the named supérieur lacks real `achat/demande-achat` access — routing target *is* the authorization for `validerDemandeAchatDirecteur()`/`refuserDemandeAchatDirecteur()`, no extra sous-module gate layered on top.

**`for_type`/`for_id` and the delivery-place fields, kept as two independent pairs.** `forType: SERVICE | CHANTIER` with `forServiceModuleCode: String?` (a bare `Module.code` value, e.g. `"rh"` — deliberately a free string, never an FK, exactly mirroring `DemandeIndex.typeModule`'s own "no FK to the UI/access referential" convention) or `projetId?`/`chantierLibre?` (the same nullable-FK-plus-free-text-fallback pair `DemandeMateriel`, Lot 5, already established). The physical delivery place is a **separate** `lieuLivraisonProjetId?`/`lieuLivraisonLibre?` pair — a request can be *for* a service department but *delivered* on a chantier, the source model's own `for_id` vs `shipping_place_id` distinction, not one field doing double duty.

**`Article` (Lot 5) gets a second, independent write path — `liste-articles.ts` is untouched.** `mettreAJourArticleFournisseurPrix()` (`achat-traitement.ts`, gated by `achat/traitement-achat`) is the only place `fournisseur`/`prix` are ever set; `creerArticle()` (Lot 5, gated by `direction-technique/liste-articles`) still only ever writes `designation`/`creeParId`. Demandeurs read Article through a **dedicated, separate query** — `listerArticlesPourDemandeAchat()` (`achat-demandes.ts`), a `select: { id, designation }`-only Prisma call — never `listerArticles()`, which exposes both price fields; this is the server-side enforcement (never trusted to the client) that demandeurs see designations only, the same principle as the RH-category restriction on document archiving (Lot 2).

**`BonEntreeMagasin` (Lot 6) gets one new optional field, `bonDeCommandeId`, wired into its existing UI — the Lot 6 circuit itself is otherwise untouched.** Same nullable shape as the pre-existing `demandeReapprovisionnementId`. `flux-entree/page.tsx`'s `BonEntreeForm` gained an optional "Bon de Commande d'origine" selector, populated by `listerBonsDeCommandePourReception()` (`achat-bons-commande.ts`, gated by `logistique/flux-entree` — a deliberate cross-module read, same shape as `resoudreLogisticien()`'s existing cross-module resolution) listing only `ENVOYE` BCs, since an `OUVERT` one hasn't been sent to a supplier yet. `MouvementStock` still has no generic origin-reference field (accepted pre-existing limitation, Lot 6) — traceability from a stock movement back to its originating BC only goes one hop, through `BonEntreeMagasin.bonDeCommandeId`.

**`BC_EMIS` is a terminal state for this lot, awaiting Lot 8 (DFC) — same "refuse/stay explicit" principle as Mission→DFC (Lot 4) and Seuil d'alerte→Achat (Lot 6).** `recalculerStatutDemandeAchat()`/`appliquerTraitementAchat()`'s urgent branch both resolve `DemandeIndex` to both validation fields `null` ("traité") at `BC_EMIS`, with an explicit code comment noting Lot 8 will pick BCs back up at this exact status for facturation/paiement — the fields captured at treatment time (`tauxTva`, `typePaiement`, `echeancePaiementJours`, `dateLivraisonPrevue`) are preparatory only, never executed in this lot. `TypeDemandeAchat` (`INITIALE`/`REGULARISATION`) is likewise a pure classification field this lot — no circuit branch reads it, a deliberate minimal-scope call, same spirit as `DemandeAchat.type`'s counterpart fields elsewhere in the app that stay inert until a later lot needs them.

**`ParametresAchat` — a single-row settings table, not a key/value store.** No key/value settings pattern exists anywhere else in this codebase to reuse; a one-row table (`seuilUrgence: Decimal`) matches the existing "table rather than enum" philosophy (`CategorieMateriel`/`UniteMesure`), correctable via `pnpm db:seed` without a migration. `obtenirParametresAchat()` does `findFirst() ?? create(...)` defensively even though seed always creates the row. The "Urgent" checkbox is re-verified server-side against this threshold in `appliquerTraitementAchat()` — never trusted from the client, same principle as the Article price-masking check.

**Seed additions**: new Fonctions "Responsable Achat" (grants `demande-achat`/`traitement-achat`/`suivi-demandes` — not `validations-parallele`, since Achat assigns validators rather than being one) and "Responsable DFC" (the first Fonction to grant real `dfc` access at all — previously only RH's catch-all incidentally included it; also granted `achat/validations-parallele`). `direction-technique` and `direction-generale`'s existing Fonctions were each declaratively extended with `achat/validations-parallele` too (not a `getModulesAccessibles()` special-case — preserves the "sidebar never special-cases" invariant). **Known, not patched**: RH's catch-all (`tousLesSousModulesSauf(["pointage"])`) also incidentally grants all 4 new achat sous-modules — same standing "not fixed as a side effect of an unrelated lot" policy as every prior lot's RH-catch-all note.

**"Suivi des demandes" is read-only, assembled from existing data, no new write path.** `listerSuiviDemandesAchat()` (`achat-suivi.ts`) does one query nesting `validations`, and `lignes.ligneBC.bonDeCommande.bonsEntreeMagasin`, letting the page compute all 7 timeline steps (soumis → validation département → traitement Achat → validation parallèle → BC émis → réception Logistique → facturation DFC) purely by inspecting already-persisted fields — step 5/6 correctly list **multiple** BC/BEM references per demande where applicable, since a demande's lines can fan out to different fournisseurs/BCs. Rendered via the existing `Accordion` component (one `AccordionItem` per demande), the same primitive already used for the per-user access matrix on the account-detail page.

### Finances et Comptabilité (`dfc` module, Lot 8)

Closes the two dependencies deliberately left terminal by earlier lots: `DemandeMission` (Lot 4, ends at `VALIDEE_RH` with an optional `fraisDeclares`) and `BonDeCommande` (Lot 7, ends at `BC_EMIS`). Replaces the two seed-only sous-modules (`paiement-standard`, `paiement-urgent-wave`) with real circuits. No new Fonction was needed — "Responsable DFC" (created in Lot 7) already granted both sous-modules.

**⚠️ Paiement Urgent Wave is entirely SIMULATED — there is no real Wave API integration in this project. ⚠️** `appellerApiWaveSimulee()` (`src/lib/paiement-urgent-wave.ts`) makes **zero network calls of any kind** — it only `console.log`s an explicit `"[SIMULATION] Appel API Wave — API non connectée..."` line and returns a fake success object whose `referenceSimulee` is permanently prefixed `SIMULATION-`, specifically so it can never be mistaken for a real transaction reference, including years later in the database. The function is deliberately **synchronous** (no `Promise`) — a real integration would necessarily be async, so this makes the difference structurally visible at every call site, not just in a comment. **Any future session wiring this up to a real Wave account must treat it as a brand-new security review** (API key management, webhook confirmation, provider-side idempotence, compliance) — never an incremental edit of the existing stub. Do not remove this warning until a real integration has actually been reviewed and shipped.

**`Fournisseur` — a new referential, linked transparently, without touching Lot 7's shipped BC-grouping logic.** `BonDeCommande.fournisseur` (a free-text `String`) is also the exact grouping key for Lot 7's already-verified `@@unique([fournisseur, ouvertMarqueur])` "one open BC per supplier" mechanism — that string field and its matching logic are untouched. Instead, `trouverOuCreerBonDeCommandeOuvert()` (`achat-bons-commande.ts`) gained a purely *additive* side effect: it also upserts a `Fournisseur` row by exact name match and sets `BonDeCommande.fournisseurId` (relation field named `fournisseurRef` in Prisma, deliberately not `fournisseur`, to avoid ever confusing it with the pre-existing string column) if not already linked. The Responsable Achat's workflow is completely unchanged — still just typing a free-text supplier name; every BC ends up linked to a real `Fournisseur` row automatically, and DFC edits that row's `numeroWave` (on the `paiement-standard` page) before attempting a mobile-money payment. `LigneDemandeAchat.fournisseur` and `Article.fournisseur` stay untouched free-text fields — only the BC header, where payment actually happens, needed the link. Test `Fournisseur` rows seeded with `numeroWave` pre-filled are deliberately disconnected from any real BC unless a tester's typed fournisseur string matches their `nom` exactly (case-sensitive) — otherwise a distinct, blank `Fournisseur` auto-creates instead.

**One `Paiement` model with two nullable FKs (`factureId?`, `demandeMissionId?`), not two duplicate models and not a polymorphic discriminant.** Exactly one of the two is set, enforced in application code — the same "sum type via two real Prisma relations" idiom already used for `DemandeMateriel.projetId?`/`chantierLibre?`. The `entiteType`+`entiteId` polymorphic pattern (`HistoriqueStatut`/`DemandeIndex`/`Notification`) is reserved for genuinely open-ended tables that any future module can write to — `Paiement` has exactly two fixed, known parent kinds, so two real relations is the correct modeling here, not a workaround. `Paiement` has no mutable `statut` field: its creation *is* the execution (DFC "exécute" in one step — the BC/Mission already passed their own full approval circuits earlier, nothing left to re-validate).

**BC facturation/payment eligibility requires `BonEntreeMagasin.statut === "VALIDE"` specifically, not mere existence of a BEM row** — this is the model's own stated semantics ("c'est la validation, pas la création, qui met à jour la fiche inventaire", Lot 6) applied consistently: a Chef Magasin's self-registered `EN_ATTENTE_VALIDATION` entry is not yet a confirmed reception. This also **fixes a real looseness in Lot 7's `suivi-demandes` page**, whose step 6 previously only checked for BEM existence regardless of status — corrected in the same pass, and step 7 ("Facturation DFC") stopped being a hardcoded placeholder, now reading the linked `Facture`/`Paiement` to report "Facturée"/"Payée"/"En attente de facturation".

**A BC can aggregate lines from multiple `DemandeAchat` rows (Lot 7's own grouping-by-fournisseur design) — paying it re-upserts `DemandeIndex` for *every* one of them, not a single row.** `executerPaiementFacture()` walks `Facture.bonDeCommande.lignes[].ligneDemandeAchat.demande`, de-duplicates by demande id, and re-`upsertDemandeIndex()`s each one with the *same* `entiteType`/`entiteId` (both validation fields stay `null` — still "traité", only `statutLibelle` changes to `"Validée — Bon de Commande payé"`, worded to land in `tonaliteDepuisStatutLibelle()`'s "succes" bucket via the substring "valid"). Verified directly: a BC grouping 2 separate demandes for the same fournisseur correctly surfaces both when paid.

**Mission payments never captured a payment mode at creation — a deliberate, documented asymmetry vs. the BC track.** `DemandeMission` has no `typePaiement` field (unlike `DemandeAchat`, which fixes it at Lot 7 treatment time), so `executerPaiementMission()` lets DFC choose the mode at payment-recording time instead of defaulting from anything — the UI explains this difference in plain text rather than pretending it's the same shape as the BC flow. Both payment paths derive `montant` from the already-recorded amount (`Facture.montant` / `DemandeMission.fraisDeclares`) server-side — never re-entered or overridable by the client, to prevent any drift between the recorded amount and the executed one.

**The Wave authorization-code circuit (`CodeAutorisationPaiementUrgent`) is the first time-boxed authorization token in this codebase — no prior pattern existed to copy.** DFC requests a code (`demanderCodeAutorisation()`, refuses if a live request/code already exists for that user, avoiding pile-up) → broadcasts to Direction Générale via the standard `enAttenteValidationDe: "DIRECTEUR"` sentinel (same shape as `appel-offres.ts`'s DG step — any real `direction-generale/validations-centralisees` holder can act, gated by the existing `peutValiderDirectionGenerale()`, never `niveauHierarchique` directly) → `validerCodeAutorisation()` is the moment the 6-digit code is actually generated (`randomInt(100000, 1000000)`, the same `node:crypto` precedent already used by `mot-de-passe-temporaire.ts`) and `dateExpiration` is set to `+24h`. **Expiry is checked at read/consumption time, never by a scheduled job** — this project has no cron infrastructure anywhere, so `EXPIRE` as a `statut` value is written only opportunistically (when `demanderCodeAutorisation()`'s anti-pileup check happens to notice a stale `VALIDE`-but-expired row) and is purely cosmetic; `executerPaiementUrgent()` and every other consumer independently re-check `dateExpiration > now()` regardless of what `statut` currently says. The code is consumed **atomically before** `PaiementUrgent` is created (`updateMany({ where: { id, statut: "VALIDE" } })`, `count === 1` guard) — this ordering, not just the guard itself, is what makes "reusing a consumed code must fail" hold under a race between two near-simultaneous attempts.

**`PaiementUrgent` is fully self-standing, matching "hors circuit standard" literally** — no `factureId`/`demandeMissionId` link of any kind, just a beneficiary (`Utilisateur` OR `Fournisseur`, same two-nullable-FK exclusive pattern as `Paiement`), `montant`, the consumed code, and `referenceSimulee`. It has no mutable `statut` either — its mere existence already means "executed", so no edit/cancel action is ever rendered for one, consistent with the circuit's explicit irreversibility requirement. The confirmation screen (récapitulatif of montant/bénéficiaire/code behind a mandatory checkbox gating the submit button) is a pure client-side UX safeguard — the server independently re-validates the code, its expiry, and the beneficiary's `numeroWave` regardless of what the client displayed, the same "never trust the client" posture already established for the Article price-masking check (Lot 7).

**`numeroWave` on `Utilisateur`** joins the existing "Profil RH" card (`gestion-comptes-acces/[utilisateurId]`, `mettreAJourProfilRH()`) rather than a new page — required for any employee/ouvrier/sous-traitant to be an eligible beneficiary of a mobile-money payment (standard `MOBILE_MONEY` or Wave urgent), checked and refused with a clear message wherever absent, never fabricated.

**`possedeDonneesLiees()` (`gestion-comptes.ts`) extended with 6 new counts** for this lot's non-cascading `Utilisateur` FKs (`Facture.enregistreParId`, `Paiement.executeParId`, `CodeAutorisationPaiementUrgent.demandeurId`/`valideParId`, `PaiementUrgent.beneficiaireUtilisateurId`/`executeParId`) — none of Lot 8's new FKs cascade, since a `Facture`/`Paiement`/`CodeAutorisationPaiementUrgent`/`PaiementUrgent` row is itself a permanent financial/audit record, same posture as `HistoriqueStatut`/`MouvementStock`. Note: Lot 7's own `DemandeAchat`/`ValidationDemandeAchat`/`BonDeCommande` FKs were never added to this function — a pre-existing gap, left as-is here since fixing it wasn't this lot's charter.

### Service QHSE (`qhse` module, Lot 9)

The first lot grounded directly in a real internal document (`Workflow_QHSE_ITA_COMPLET_v2.docx`, supplied mid-planning alongside `ORGANIGRAMME VERSION 02.pdf`) transcribed faithfully rather than paraphrased from a cadrage — every field name across the 9 flows below is verbatim from that source. Builds all 9 already-seeded `qhse` sous-modules: `accueil-securite`, `ast`, `stock-epi`, `inspection-hse`, `rapport-hebdo`, `programme-sensibilisation`, `pv-sensibilisation`, `rapport-incident`, `non-conformite`. One new Fonction, **"Responsable QHSE"** (all 9 sous-modules — the org chart's "Assistant QHSE"/"Relais QHSE" sub-positions are real access, not separate Fonctions, following Lot 6's precedent of bundling several document roles into few Fonctions), test account `qhse.test@itamanager.cloud`.

**Stock EPI (Flux 3) reuses Lot 6's `Materiel`/`CategorieMateriel`/`MouvementStock` machinery — a filtered read, not a duplicate stock model — but EPI *issuance* is nominative, which the initial no-nominative-tracking assumption got wrong until the real document was read.** The "Fiche de Gestion du Stock des EPI" form is explicitly per-beneficiary (Nom/Prénoms, Qté Sortie, Retour), so a new `AttributionEPI` model exists (beneficiary `Utilisateur`, `Materiel`, quantité, `retourEpiUsage`, `mouvementStockId? @unique` tracing back to the `MouvementStock` row it produced — same reverse-trace idiom as `BonEntreeMagasin.demandeReapprovisionnementId`). Its stock decrement reuses the exact atomic conditional-update pattern from `flux-sortie.ts`'s `deciderDMS()` verbatim: an interactive `$transaction`, `materiel.updateMany({ where: { id, quantiteStock: { gte } } })`, `count === 0` → refuse the overdraft with no partial writes. `qhse/stock-epi` is a self-contained sous-module (its own `requireAccesModule` guard), independent of Logistique's Fonction/access wiring — filters `Materiel` by `categorie.code === "EPI"` at read time.

**Flux 1 (Accueil Sécurité) is triggered by account creation — but `prisma/seed.ts`'s `seedUtilisateursTest()` has its own fully independent inline user-creation logic that never calls `creerCompteUtilisateur()`/`initialiserAccesDepuisFonction()`, so the hook had to be wired into both places or every seed test account would silently end up untestable for this lot.** `initialiserAccueilSecurite(utilisateurId)` (`user-provisioning.ts`, a one-liner: `prisma.accueilSecurite.create({ data: { utilisateurId } })`) is called from `creerCompteUtilisateur()` right after `prisma.utilisateur.create()`, wrapped in the exact same defensive try/catch shape CLAUDE.md documents for the welcome email (log, never rethrow, never block account creation on failure) — and inlined directly (not imported — `seed.ts` can't import this `server-only`-tainted module, same documented limitation as `propagerModificationFonction()`) at the equivalent point in `seedUtilisateursTest()`'s loop, for every seeded account. Verified end-to-end: a fresh account (real or seeded) always gets exactly one `AccueilSecurite` row at `EN_ATTENTE`, never duplicated on retry.

**Non-Conformité (Flux 7) is a shared arrival point from two independent origins, modeled identically to Lot 6's `creerOuReutiliserDemandeReapprovisionnement()`.** `creerOuReutiliserNonConformite()` (`qhse-non-conformite.ts`) takes a discriminated union (`{ type: "POINT_INSPECTION_HSE", reponsePointId, libellePoint }` or `{ type: "RAPPORT_INCIDENT", rapportIncidentId, description }`), `findFirst`s on whichever origin FK is non-null, creates if absent. `NonConformite` has two nullable FK columns, `reponsePointInspectionHSEId? @unique` and `rapportIncidentId? @unique` — Postgres treats `NULL` as always-distinct in a unique index, so the two origins (and manually-created rows, both null) never collide. Verified directly: an Inspection HSE `NON` point and a Rapport Incident's `nonConformiteIdentifiee` flag each produce their own `NonConformite`, re-triggering the same origin is idempotent (no duplicate), and clearing/closing one never touches the other.

**Rapport Incident's Non-Conformité trigger (`nonConformiteIdentifiee: Boolean`/`nonConformiteDescription?`) is a deliberate app-only addition, not a field lifted from the paper form.** The real "Rapport d'Incident/Accident" has no clean universal trigger for this — only a narrow checkbox buried inside one of several "causes" multi-selects — so this is a new digital-only field, documented here as such rather than silently invented.

**Inspection HSE (Flux 4A) reuses the existing generic `PointInspection` referential (Lot 6) rather than a new table** — `CategorieInspection` gained a third value, `HSE`, alongside `VEHICULE`/`ENGIN`. The 26 points (Base vie ×8, Poste de travail ×5, Sensibilisation & EPI ×9, Personnel chantier ×2, Gestion déchets ×2) are **real content transcribed from the source document**, not placeholder — unlike Lot 6's `PointInspection` seed, which explicitly flagged most of its véhicule/engin rows as plausible-but-invented filler pending real content. `ReponsePointInspectionHSE` uses a new `enum ReponseInspectionHSE { OUI NON }` — a compliance yes/no, deliberately kept separate from the existing `EtatPointInspection` condition-rating enum since the two questions ("is this compliant" vs. "what condition is this in") aren't the same axis. A `NON` answer triggers `creerOuReutiliserNonConformite()` per point, inline in `creerInspectionHSE()` after the nested `reponsesPoints` create.

**Programme/Séance/PV de Sensibilisation (Flux 5A/5B) is a 3-level structure, not 2 — corrected mid-lot after the real document showed a period header containing a multi-row table of planned séances (Date/Thème/Animateur/Commentaire), contradicting an earlier answer given before that document was read.** `ProgrammeSensibilisation` (period header) → `SeanceSensibilisation[]` (cascades, one row per planned session) → `PVSensibilisation` (Flux 5B, `seanceId @unique`, does **not** cascade — a signed PV is a permanent record, same posture as every other terminal QHSE form) → `ParticipantPVSensibilisation[]` (cascades, free-text nom/poste/aSigne — not necessarily existing `Utilisateur` rows, since sensibilisation sessions can include subcontractors/visitors). `creerPVSensibilisation()` checks for an existing PV on that séance first and returns a clean application error rather than surfacing a raw unique-constraint violation.

**AST (Flux 2) validation needs an explicit self-validation guard beyond the usual sous-module check, because both submitter and validator Fonctions hold the same access.** "Chef de chantier" and "Responsable QHSE" both have `qhse/ast` — without an extra `acteurId !== fiche.chefChantierId` check in `validerAST()`, a submitter could approve their own AST. This mirrors the reasoning (not the mechanism) behind Lot 5's DT contre-proposition acceptance being routed to the original `initiateurId`.

**`possedeDonneesLiees()` (`gestion-comptes.ts`) extended with ~20 new counts**, one per new non-cascading `Utilisateur` FK across all 9 models (`AccueilSecurite.responsableAccueilId`, `FicheAST.chefChantierId`/`relaisQHSEId`, `AttributionEPI.beneficiaireId`/`effectueParId`, `InspectionHSE.responsableInspectionId`/`relaisQHSEId`/`chefChantierId`, `RapportHebdoQHSE.relaisQHSEId`, `RapportIncident.reporteParId`/`creeParId`, `NonConformite.identificateurId`/`responsableQHSEId`, plus the `AccueilSecurite.utilisateurId` 1:1 itself) — every one of these is a permanent compliance/audit record, same posture as `HistoriqueStatut`/`MouvementStock`, never cascading from `Utilisateur`. Learned from Lot 8's own documented gap (its FKs were added here, but Lot 7's were not, and that was never retroactively fixed) — this lot's extension was done proactively rather than left as a second gap to rediscover later.

### Dependency-tree quirk: keep zod pinned to one version

`pnpm-workspace.yaml` has `overrides: zod: ^4.4.3`. Without it, the `shadcn` CLI (a devDependency) transitively pulls in an old `zod@3.25.x` via `@modelcontextprotocol/sdk`, and pnpm's strict linking resolves `@hookform/resolvers/zod` against that old copy instead of the top-level zod — this breaks `zodResolver()`'s TypeScript overloads in `login-form.tsx`/`change-password-form.tsx` with a confusing "Types of property '_def' are incompatible" error even though both are nominally "zod v4". If a fresh `pnpm install` ever reintroduces a duplicate zod, check `pnpm why zod` before debugging the forms themselves.

That same override breaks the `shadcn` CLI itself at runtime: `shadcn` declares `zod: ^3.24.1` and calls `.deepPartial()` internally, a method zod v4 removed — `npx shadcn add <component>` crashes with `TypeError: s.deepPartial is not a function` because `npx` resolves `zod` from this project's (overridden) `node_modules`. Use `pnpm dlx shadcn@latest add <component>` instead — it runs in an isolated temp install outside this project's `pnpm-workspace.yaml`, unaffected by the override.

### Prisma version note

Pinned to Prisma **6.x** deliberately, not the latest 7.x. Prisma 7 removed `datasource.url`/`directUrl` from `schema.prisma` in favor of driver adapters configured in `prisma.config.ts`, which is a bigger migration than this project needs right now (Supabase's standard pooled/direct URL split works out of the box on v6). Don't casually bump past 7 without accounting for that.
